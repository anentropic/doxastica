### Central-Trap Verification (cut-then-max REWIND, not max-then-drop)

The PLAN identifies the central trap of the phase as conflating the `<= as_of` PRE-filter (cut-then-max = REWIND) with `query_scope`'s `event_id_max` POST-filter (max-then-filter = DROP).

**Implementation verified:** `core.py` lines 657-662:
```
for row in rows:
    if row["source_event_id"] > as_of:  # D-04 inclusive cut: keep <= as_of, drop newer
        continue
    current = by_belief.get(row["belief_id"])
    if current is None or _order_key(row) > _order_key(current):
        by_belief[row["belief_id"]] = row  # cut-window status-agnostic tail (D-05 tiebreak)
```
The cut `> as_of` is inside the group-by loop as a PRE-filter, before the per-group max. This is precisely cut-then-max (REWIND). Compare with `query_scope` lines 599-604 which apply `event_id_max` as a POST-filter on already-derived tails (DROP). The trap is not present.

**Test guard:** `test_cut_rewinds_to_older_value` asserts `_ids(result) == {"A", "B"}` and `_value_of(result, "B") == "w_old"` at the older-event cut, proving the REWIND. Both backends pass.

### Data-Flow Trace (Level 4)

`get_scope_at` is a pure read method — it reconstructs from immutable nodes and returns frozen `BeliefState` models. The data flow:

1. `match_nodes("BeliefState", {"scope_id": scope_id})` — fetches all raw dicts from backend storage.
2. Cut-then-max loop — derives the per-belief cut-window tail in-core (no backend call inside the loop).
3. Retracted collapse — drops beliefs whose winning status is `retracted`.
4. `_hydrate(t)` — decodes the base64-JSON value token and builds a frozen `BeliefState` model.

The 7 example tests assert on concrete values (`_value_of(result, "B") == "w_old"`, etc.) on BOTH backends, proving real data is flowing from storage through hydration. No hollow prop or empty-array stub pattern present.
